# 302 跳转服务

302 跳转服务是系统的核心组件，负责将 STRM 文件中的 pickcode 解析为 115 CDN 的媒体下载地址。

## 为什么需要 302 跳转？

115 网盘的媒体文件不提供永久直链。每次播放都需要通过 115 API，使用文件的 pickcode 获取一个带时效的 CDN 下载 URL。这个 URL 有以下几个特点：

1. **时效性** — URL 通常在数分钟后过期
2. **UA 绑定** — URL 与请求时的 User-Agent 绑定，不同 UA 拿到不同 URL
3. **区域限制** — 某些 URL 有 IP 限制

如果直接在 STRM 文件中写入静态 URL，URL 过期后播放会失败。因此 STRM 文件中写入的是跳转服务自己的地址，播放时实时请求跳转服务获取最新的 CDN URL。

## 架构演变

### 版本 1：纯 302 重定向

跳转服务返回 HTTP 302 响应，`Location` 头指向 115 CDN URL。客户端跟随跳转直接访问 CDN。

**问题**：Web 浏览器跨域（CORS）导致播放失败。

### 版本 2：流式代理（当前）

跳转服务不再返回 302，而是由反向代理的 `_stream_from_cdn` 方法用 httpx 从 CDN 拉取媒体数据，逐块流式返回客户端。

**优势**：
- 客户端全程只连代理，不直接访问 CDN
- CORS 问题完全消除
- 客户端原始 UA 和 Range 头自动转发，满足 UA 绑定和拖放进度

## 核心机制

### UA 绑定的加密下载 API

使用 `p115rsacipher` 库对 `{"pick_code": "..."}` JSON 进行 RSA 加密，POST 到 `proapi.115.com/android/2.0/ufile/download`，携带客户端的 User-Agent。返回数据经解密后得到 CDN URL 和过期时间戳。

### 基于 UA 的缓存

缓存 key = `pickcode:sha256(user_agent)[:16]`，TTL = CDN URL 过期时间 - 300 秒（留余量），最大 90 秒。最多缓存 1000 个条目。

### 文件名的国际化处理

CDN URL 中的文件名可能包含非 ASCII 字符，采用 `Content-Disposition: attachment; filename*=UTF-8''encoded_name` 格式确保浏览器正确识别。
